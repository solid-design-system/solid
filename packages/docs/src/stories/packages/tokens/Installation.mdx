# Installation

  The `components`, `styles`, and `tokens` packages now always share the same version. We use fixed versioning to keep

## Prerequisites

### Fonts

Solid doesn't provide **any fonts**, you have to include them yourself.
To do this, you can either use a CDN (our recomendation) or self-host the font files.
Please reach out to the brand department if you want to self-host the font.

#### **Using a CDN**

The following CSS gives you a jump start on how to set up fonts using a CDN.

#### **Self-host Fonts**

To self-host fonts, reach out to your brand department to obtain the woff2 files (or woff/ttf).
Then, add these files to a public folder (e.g. fonts) within your project.

<sd-accordion summary="Example CSS for Self-Host Fonts">

```css
/** Genos GFG Font Faces for VB Theme **/
@font-face {
  font-family: 'Genos GFG';
  font-style: normal;
  font-weight: 400;
  font-display: swap;
  src: url('/fonts/geno/GenosGFG-Regular.woff2') format('woff2');
}

@font-face {
  font-family: 'Genos GFG';
  font-style: normal;
  font-weight: 700;
  font-display: swap;
  src: url('/fonts/geno/GenosGFG-Bold.woff2') format('woff2');

.body {
  font-family:
    'Genos GFG',
    ui-sans-serif,

    BlinkMacSystemFont,

    'Noto Sans',

    'Apple Color Emoji',
    'Segoe UI Emoji',
    'Segoe UI Symbol',
    'Noto Color Emoji';
}
```

</sd-accordion>

#### **Improve Font Rendering**

To improve font rendering, the following CSS has to be added to your project as well:

<sd-accordion summary="Example CSS">

```css
/**
  * Render font correctly.
  */

:root {
  font-synthesis: none;
  text-rendering: optimizeLegibility;
  -webkit-font-smoothing: antialiased;
  -moz-osx-font-smoothing: grayscale;
  -webkit-text-size-adjust: 100%;
  line-height: 1.5;
  font-size: 16px;
}
```

</sd-accordion>

### Normalizing browser styles

We recommend using [normalize.css](https://necolas.github.io/normalize.css/) or similar.
This will handle cross-browser inconsistencies and set furthermore some improvements,
e. g. regarding improved line-heights for `<sub>` and `<sup>` elements.
If you don't want to use normalize.css, you should at least reset
the line-heights for `<sub>` and `<sup>` elements, as shown below.

<sd-accordion summary="Example CSS">

```css
/**
  * Prevent `sub` and `sup` elements from affecting the line height in
  * all browsers.
  */

sub,
sup {
  font-size: 75%;
  line-height: 0;
  position: relative;
  vertical-align: baseline;
}

sub {
  bottom: -0.25em;
}

sup {
  top: -0.5em;
}
```

</sd-accordion>

## Themes

Themes allow you to easily switch between predefined themes (such as light or dark mode, or brand-specific) and even create your own custom themes by overriding CSS variables.

### Importing themes

> This only has to be done once per document, every application and component using Solid Components/Styles on that page can then automatically use the included themes.

Each theme is defined in a CSS file containing variables.
You must import at least one theme to use Solid Components or Styles.

### Switching a theme

#### User Setting (Light/Dark Mode)

If you just want to respect the user's system preference for light or dark mode, you can adaptively load the appropriate theme based on the `prefers-color-scheme` media query.

```html
<link
  rel="stylesheet"
  href="https://solid-design-system.fe.union-investment.de/tokens/%TOKENS-VERSION%/cdn/themes/ui-light/ui-light.css"
  media="(prefers-color-scheme: light)"
/>
<link
  rel="stylesheet"
  href="https://solid-design-system.fe.union-investment.de/tokens/%TOKENS-VERSION%/cdn/themes/ui-dark/ui-dark.css"
  media="(prefers-color-scheme: dark)"
/>
```

For further information how to make this more dynamic check out [this article](https://pepelsbey.dev/articles/native-light-dark/).

#### Class-based theming (Light, Dark, Other Brands)

For finer granularity use the theme's class name and set it on the body or the desired element.

> After importing it, `ui-light` is automatically applied, if no other theme is specified.

Each theme class follows this pattern: `sd-theme-[name]`.

Example for `ui-dark`:

```html
<body class="sd-theme-ui-dark">
  <!-- Your website -->
</body>
```

This will apply the UI Dark theme globally on your website. This makes it easy to switch between themes, for example when toggling between light and dark mode.

### Creating your own theme

You can create your own theme by overriding CSS variables.
There are two main types of variables (tokens):

#### **1 - Primitive tokens**

Primitive variables define the base values used across all components.
They act as fallbacks for all other variables.

Changing the primary color globally:

```css
--sd-color-primary: 255 0 85;
```

This change will affect every component that uses the primary color.

#### **2 - Semantic tokens**

Semantic tokens are divided into general and component-specific variables.

##### **2.1 - General semantic tokens**

These let you target specific use cases without interfering with other parts that use the same primitive variable (e.g., primary background color without affecting all primary color usages).

Example:

```css
--background-color-primary: 255 230 238;
```

This overrides only the “primary background” usages, falling back to the primitive tokens otherwise.

##### **2.2 - Component-specific semantic tokens**

If you want to customize variables for a specific component only, you can override its semantic tokens.
Example, changing only the primary background color of the `sd-button` component:

```css
--sd-button--primary--default--default-color-background: 255 0 85;
```

<blockquote>The list of available component variables is documented on each component’s `docs` page.</blockquote>

**Note:** Even if a component doesn’t yet have full semantic variable support, you can still apply overrides within the component’s scope using general semantic tokens.

Example:

```css
sd-button {
  --background-color-primary: 255 0 85;
}
```

This will only affect the primary background color inside `sd-button` components.

## Tokens

You can use design tokens to build own components and websites. The following solutions are provided:

- Vanilla CSS (via the provided CSS variables)
- SCSS
- TailwindCSS (v3)
- TailwindCSS (v4)

> This is still Work in Progress.

### Vanilla CSS

Import a theme as described above and use the CSS variables in your own styles.

Example:

```css
.my-special-element {
  background-color: rgba(var(--sd-color-background-primary-100));
  color: rgba(var(--sd-color-text-black));
}
```

### SCSS

Import the SCSS files from `@solid-design-system/tokens` and use the SCSS variables in your own styles.

```scss
@import '@solid-design-system/tokens/dist/scss';
```

Example:

```scss
.my-special-element {
  background-color: rgba($sd-color-background-primary-100);
  color: rgba($sd-color-text-black);
}
```

### TailwindCSS (v4)

Import the TailwindCSS configuration from `@solid-design-system/tokens` and extend your own Tailwind configuration with it.

```css
@import 'tailwindcss';
@import '@solid-design-system/tokens/dist/themes/tailwind.css';
```

This will automatically include all Solid Design System tokens into your Tailwind setup and make them available as classes.

### TailwindCSS (v3)

Import the TailwindCSS configuration from `@solid-design-system/tokens` and extend your own Tailwind configuration with it.

```js
// tailwind.config.js
import solidTheme from '@solid-design-system/tokens/dist/tailwind.json';

export default {
  theme: {
    extend: {
      ...solidTheme
    }
  }
};
```

### Minimum dependency requirement

Currently, any version of the **components** and **styles** packages is compatible with any version of the **tokens** package.

In a future release:

- A **minimum dependency requirement** will be introduced
- The **tokens package version must be equal to or higher than** the components and styles versions

We recommend reviewing your setup now to ensure that your tokens package is **aligned with (or ahead of)** your components and styles. This will help you avoid compatibility issues once the new requirement is enforced.
